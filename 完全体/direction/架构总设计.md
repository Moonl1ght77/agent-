## 一、结论先行（三个核心判断）

### 判断 1：M1-M7 全拆分 → 过度工程，定为「3 分析 Agent + 1 总控编译器」

素材B 猜想的 M1-M7 逐项拆分，每项一次 LLM 调用，代价是：串行延迟 ×7、变量传递损耗、
每层独立幻觉风险。而素材A（V8.2 已验证可跑）证明 GPT-5.5 一次读 6 图就能完成全部分析。
但素材A 走了另一个极端——单节点 15k 字符提示词把「事实提取」和「提示词编译」混在一起，
出错时无法定位、无法人工干预。

**完全体取中间值**（这正是那张 85% 参考图有效的原因——锁卡显式化、可人工检查）：

| 完全体节点 | 吸收素材B的 | 输出 |
|---|---|---|
| A1 商品DNA锁定官 | M1 事实 + M2 潘通色 + C类衣服锁8层 | GARMENT_DNA 锁卡（8字段） |
| A2 模特DNA与造型官 | M3 模特DNA + M5 穿搭 + 身材需求卡 | MODEL_DNA + STYLING_PLAN |
| A3 场景与八分镜官 | M4 背景光影 + M7 镜头 + G类机位锁 | SCENE_DNA + SHOT_PLAN_8 |
| C 总控编译器 | 素材B「总控+中文编译器」+ 素材A TEXT1 骨架 | 8 组英文正/负提示词 |

调用次数 8→4，同层并发（A1/A2/A3 平台自动并行），且每张锁卡都是**可读、可改、可复用**
的中间产物——商用可控性的关键。

### 判断 2：C类锁 / G类锁 / 级联控制线 → 用「锁卡文本 + 原生多图输入」优雅替代

- **C类衣服锁**不需要 8 个控制节点：它是 A1 输出的 8 字段结构化文本，注入总控。
- **G类机位锁**不需要 8 个锁节点：它是 A3 输出的 SHOT_PLAN_8 表 + 总控内置分镜导演规则
  （素材A 的 Six-Set Shot Director 已验证此法有效，完全体扩到 8 位）。
- **真正的锁发生在生图卡**：GPT-Image 2 / Gemini 3 原生多图输入——每张生图卡按固定
  槽位挂参考图（槽1模特/槽2商品/槽3细节/槽4配饰资产/槽5场景），提示词以
  "Input Image N is ..." 角色映射前缀开头（素材A 验证过的 Visual Prompting 写法）。
  平台规则「连线顺序 = 参考图槽位顺序」（见 docs/prompting.md），所以**连线顺序即锁**。

### 判断 3：不是复刻，是「骨架继承 + 缺口补全」

素材A 的骨架（总控→抽取→原生多图生图→QA）已被验证，予以继承；但它有实锤缺陷
（见第二节），补上这些缺口后才是完全体。85% 参考图的增量优势（需求卡、锁卡显式化、
详情页组合）全部吸收进来。

## 二、素材A（V8.2）解析结果与实锤缺陷清单

### 2.1 拓扑还原（26 卡 / 69 连接）

```
输入图1模特 ─┬─────────────────────────────┐
输入图2商品 ─┼→ TEXT1(总控C,15k字符) → TEXT2..7(P抽取×6) → IMAGE2..7(生图×6) → QA质检
输入图3细节 ─┤        ↑                                    ↑ (每张挂图1/2/3/4/5共5参考图)
输入图6动作 ─┘        │
输入图2/3/6 → TEXT8(选品) → 输入图4·白底资产(生成)──┘
输入图5场景原图 → 场景清洗(去人去水印) = 输入图5场景参考图 ──┘
IMAGE3/4/6+图2/3 → 9:16详情页 ； 图2/3 → 四宫格细节图
```

### 2.2 实锤缺陷（完全体逐条修复）

| # | 缺陷 | 证据 | 完全体修复 |
|---|---|---|---|
| 1 | **参考图槽位与提示词声明错位**：TEXT8 连线序为图6,图3,图2，提示词却声明 Image1=商品/Image2=细节/Image3=风格；IMAGE4 连线序为商品,细节,场景,资产,模特，TEXT1 前缀却声明 1=模特/2=商品…GPT-5.5 靠语义自愈，但这是随机劣化源 | canvas.json connections 顺序 vs 提示词原文 | 构建器强制「连线顺序=前缀声明顺序」，一处定义、程序保证 |
| 2 | **缺背面机位**：TEXT1 Set4 明确 "not back or strong side"，全流程无背面图——服装电商详情页刚需 | TEXT1 §15 | 新增 P7 背面结构位（后领/背部版型/背部印花） |
| 3 | **姿势图没有真正喂给生图模型**：所有 IMAGE 卡只挂 5 图，但前缀声明了 6 图（Input Image 6 只存在于文字里）| IMAGE 卡入边清单 | G1 真挂第 6 张姿势图；Set1 用 6 图前缀，Set2-8 用 5 图前缀，声明与实际严格一致 |
| 4 | **无用户需求卡**：品类/风格/身材全靠改 15k 字符提示词 | 全画布无 text 卡 | 需求卡①②（text 卡），只改卡不动提示词 |
| 5 | **QA 是死胡同**：REPAIR_BRIEF 无下游，返修靠人肉重跑 | QA 无出边 | 新增 R 返修编译器 + RG 返修生图卡（单卡重跑，规避平台禁环） |
| 6 | **详情页节点简陋**：整句提示词只有"根据图片生成这件衣服的电商详情页"，且 AI 生成中文文字必乱码 | card_20 | X2 结构化详情页首屏提示词 + 「禁止文字占位」规则（文字后期排版加） |
| 7 | **商用规则缺失**：无 logo 几何锁、肤色受光真实度、暗色面料细节、条纹对花、身材真实性 | TEXT1 全文 | 十条商用规则库（见第五节）写入各锁卡与总控 |

## 三、完全体拓扑（37 卡 / 6 运行组）

```
[需求卡①品类与输出]──┬──────────────┐
[需求卡②造型与身材]──┼──────────┐   │
                      │          ↓   ↓
IN1模特 ─────────────┼────→ A2 模特DNA ─┐
IN2商品 ─┬───────────┼→ A1 商品DNA ────┼→ C 总控编译器 → P1..P8 抽取 → G1..G8 生图 → QA 质检
IN3细节 ─┤           │        ↑        │      ↑(挂6参考图核对)   ↑(每张挂5图,G1挂6图)  │
IN4场景原图→S1场景清洗┼→ A3 场景八分镜 ─┘      │                                       ↓
IN5姿势 ─────────────┘        ↑                │                              R 返修编译 → RG 返修生图
IN6穿搭风格 → S2A选品 → S2B白底资产 ───────────┘
IN2/IN3+A1 → X1 四宫格细节 ；G1/G3/G6+IN2+X1+A1 → X2 详情页首屏
```

**运行组（= 商用检查点）**：
①资产准备(S1,S2A,S2B) → ②分析与编译(A1,A2,A3,C,P1-8) → ③八分镜生图(G1-8) → ④质检(QA)
→ ⑤返修支线(R,RG，按需) → ⑥详情页(X1,X2)

组间是人工检查点：跑完②可以先看锁卡和 8 组提示词，改需求卡重跑②，满意再花钱跑③。
（平台引擎：组内拓扑分层自动并发，A1/A2/A3 并行、P1-P8 并行、G1-G8 并行。）

**八分镜定义（G类机位锁完全体）**：

| 分镜 | 机位 | 用途 |
|---|---|---|
| P1 姿势参考位 | 全身/中远景，动作几何来自 IN5 | 主图候选 |
| P2 空间深度位 | 全身，门口/走廊/光影边界 | 版型垂坠展示 |
| P3 上身细节位 | 中景半身，领口肩袖 | SKU 上身细节 |
| P4 廓形垂坠位 | 全身微侧正面 | 廓形展示 |
| P5 动态生活位 | 倚/坐/整理衣角/行走停顿 | 种草图 |
| P6 穿着细节特写 | 身上细节近景 | 面料/工艺信任感 |
| P7 背面结构位（新增） | 背面或3/4侧背全身 | 详情页刚需 |
| P8 环境叙事位（新增） | 远景，人物占比小 | 氛围种草/首图备选 |

**参考图槽位协议（全流程统一，程序保证）**：
槽1=IN1模特 ｜ 槽2=IN2商品 ｜ 槽3=IN3细节 ｜ 槽4=S2B配饰资产 ｜ 槽5=S1净化场景 ｜ 槽6=IN5姿势(仅G1)

## 五、补全的十条商用规则（素材A/B 均缺失或不全）

| # | 规则 | 落点 |
|---|---|---|
| R1 | **Logo/印花几何锁**：逐字转写、位置与相对比例锁定；负向词 distorted/warped/stretched logo, misspelled text | A1字段5 + C §商品锁 + QA检查项 |
| R2 | **肤色受光真实度**：锁定模特肤色深浅，深肤禁提亮漂白、浅肤禁过度美黑；肤色在场景色温下自然渲染（深肤保留暖调次表面反射，避免发灰） | A2字段3 + C §身份锁 + QA |
| R3 | **白色过曝 + 深色死黑双保护**：白衣保留纹理不过曝（素材A已有）；黑/深色衣必须保留褶皱与结构细节，不糊成色块（新增） | C §场景光线 + QA |
| R4 | **图案对花连续性**：条纹/格纹在缝线、口袋、门襟处走向连续，不断裂错位 | A1字段5 + C + QA |
| R5 | **面料物理行为**：垂坠/挺括/弹力贴身由 A1 判定并注入每组提示词，防止雪纺画成西装料 | A1字段3 + C |
| R6 | **身材真实性（合规）**：按需求卡身材如实出图，禁自动瘦身/拉腿/磨骨；大码即大码 | A2字段5 + C + QA |
| R7 | **手部与肢体完整性**：负向词常驻 malformed hands, extra fingers, twisted limbs | C §负向规则 |
| R8 | **背面结构一致性**：P7 专用——后领标、背部剪裁、背部印花位置与 IN2/IN3 一致；背面位不检查人脸，检查发型+体型连续性 | C §Set7 + QA |
| R9 | **输出规范**：3:4 电商主流比例；主商品完整可见不裁切关键部位；首图预留平台角标安全区 | 生图卡参数 + C |
| R10 | **抗AI感全套**（素材A 验证有效，全量继承）：iPhone 后置1x直拍、自然曝光、生活痕迹、禁影楼灯/强虚化/磨皮/网红脸；八组表情/站位/朝向不重复 | C §Realism/§Diversity |

## 六、DBS 三层解耦

```
完全体/
├── direction/    架构总设计.md              ← 方向层：决策与理由（本文件）
├── blueprints/   workflow_spec.json         ← 数据/规则层：节点、连线序、分组、布局
│   └── prompts/  *.txt                      ← 数据/规则层：全部系统提示词（与逻辑零耦合）
└── solutions/    build_lxf.py               ← 工具层：读 blueprints → 产 .lxf（离线交付）
    └──           apply_to_app.py            ← 工具层：读 blueprints → 本地API声明式搭图（在线落地）
```

改提示词 → 只动 prompts/*.txt；改结构 → 只动 workflow_spec.json；两条产出路径共用同一份
蓝图，永不分叉。这就是「逻辑层 / 数据规则层 / Python 工具层完全解耦」的落地形态。

## 七、语言策略（"中英结合生图优化版"的落地）

| 节点 | 语言 | 理由 |
|---|---|---|
| A1/A2/A3 分析官 | 中文指令 → 中文字段名+英文值 | 用户可读可改锁卡；英文值直接进生图提示词 |
| C 总控编译器 | 纯英文 | 素材A 15k 英文提示词是被验证的配方；输出必须是英文生图提示词 |
| P 抽取器 | 中文一句话 | 只做抽取，中文指令最不易被误执行 |
| QA / R | 中文输出 | 质检报告给人看 |
| 需求卡/说明卡 | 中文 | 用户填写界面 |

## 八、使用方式与 V9.1 路线

**使用**：① 导入 `V9.0完全体8分镜.lxf`（或用 apply_to_app.py 直接搭进画布）→ ② 替换 6 张
输入图 → ③ 改两张需求卡 → ④ 依次运行组①→②（此处可人工检查锁卡与8组提示词）→③→④
→ ⑤ 看 QA 报告，需要返修就跑组⑤（RG 单卡重跑）→ ⑥ 满意后跑组⑥出详情页素材。

**换渲染模型**：G1-G8 卡片把 model 从 `gpt-image-2` 改为 Gemini 3 Pro Image 对应 SKU 即可，
提示词协议（角色前缀+正负向）两家通用；Gemini 系对多图空间关系理解更强，Logo 文字保真
略弱，建议 A/B 后定主力。

**V9.1 路线（不进本版）**：棚拍白底商用风变体（换 C 卡即可，骨架不动）；ai_tryon 卡做
换装预处理；多 SKU 批量循环（Automation Gateway 脚本级循环 apply+run）；QA 分数自动
触发 RG 重跑（需网关 run 脚本轮询）。
