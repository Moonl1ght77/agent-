# 核心理念 —— DBS 三层解耦 + 九条架构判断

> Direction 层。回答"为什么这样设计"。所有判断均来自实测验证，不是理论推演。

## 一、DBS 框架（Direction / Blueprints / Solutions）

任何生图工作流项目必须按三层组织，**完全解耦**：

| 层 | 内容 | 变更规则 |
|---|---|---|
| **D**irection 方向层 | 架构决策与理由、裂变记录 | 只在架构变化时更新，记录"为什么" |
| **B**lueprints 规则层 | `workflow_spec.json`（节点/连线序/分组/布局）+ `prompts/*.txt`（全部提示词） | **唯一真源**。改提示词只动 prompts，改结构只动 spec |
| **S**olutions 工具层 | 构建器（蓝图→平台文件）、apply 脚本（蓝图→在线画布） | 纯确定性代码，程序保证一致性，禁止藏业务规则 |

**为什么解耦是硬要求：**
- 提示词与结构混在平台文件里 = 无法 diff、无法回滚、无法程序校验，人工维护必然漂移。
- 两条产出路径（离线文件 + 在线API）共用同一份蓝图，永不分叉。
- 出错时能定位到层：图不对→查 prompts；连线不对→查 spec；编译不对→查工具。

## 二、九条架构判断（每条都有翻车史或验证史）

### 判断 1：分析节点取中间值 ——「3 分析 Agent + 1 总控编译器」
- 逐项拆分（7+个分析节点）= 过度工程：串行延迟、变量传递损耗、每层独立幻觉。
- 单个 15k 字符巨型提示词 = 另一个极端：事实提取和提示词编译混在一起，出错无法定位、无法人工干预。
- **正解**：按"输出物"分组——商品事实一个 Agent、人物造型一个 Agent、场景分镜一个 Agent，各产一张**人类可读可改的锁卡**；再由一个纯编译器把锁卡编译成 N 组生图提示词。

### 判断 2：锁不是节点，锁是「结构化文本 + 原生多图槽位」
- 一致性锁定不需要专门的"锁节点"：商品锁=分析Agent输出的结构化字段；机位锁=分镜表+编译器内置导演规则。
- **真正的物理锁发生在生图卡**：生图模型原生多图输入，每张生图卡按固定槽位挂参考图，提示词以 "Input Image N is ..." 角色映射前缀开头。连线顺序=槽位顺序，**所以连线顺序即锁**。

### 判断 3：中间产物显式化 = 商用可控性
每张锁卡都是可读、可改、可复用的检查点。商家看得懂锁卡，才敢把钱花在生图上。黑盒直通的工作流无法商用。

### 判断 4：编译器/抽取器模式，杜绝改写
- 一个编译器一次产出全部 N 组正/负提示词（跨图一致性只能在单点保证）；
- 下游抽取器**只做抽取**（"原样取出第 X 组，一字不改"），中文一句话指令最不易被误执行；
- 若让每个分镜各自"理解并改写"，8 张图必然漂移。

### 判断 5：运行组 = 商用检查点（省钱结构）
把工作流切成运行组：①预处理 → ②分析编译 → ③生图 → ④质检 → ⑤返修 → ⑥组合输出。
组间是人工检查点：跑完②先看锁卡和提示词（便宜），满意再花钱跑③（贵）。**结构本身就是预算控制。**

### 判断 6：反馈做支线，不做回路
平台引擎（DAG拓扑排序）禁环。QA→生图的级联反馈必须做成**向下的返修支线**（QA→返修编译器→返修生图卡，单卡重跑），不能连回生图节点。

### 判断 7：用户心智 = 所见即所得
商家用户对输入图的理解是"我给什么你就用什么"。AI 自主选品/配饰是负资产（商家有固定搭配），删掉反而更稳更省钱。输入图语义必须直白：模特图=用这个人，商品图=画这件衣服，搭配图=就穿这件。

### 判断 8：图看不出来的属性，必须做成硬指令字段
白底平铺图无法传达弹力面料的上身松紧（实测：需求写"紧身"仍出宽松图）。凡是视觉推断不出的商品属性（松紧/垂感/弹力/厚薄/上身效果），必须：
**需求卡显式字段 → 分析Agent强制覆盖视觉推断 → 编译器每组提示词强制注入正向短语+负向词常驻 → QA 专项检查+硬失败条款**。四级贯通，缺一级就会被模型的"平均先验"吃掉。

### 判断 9：语言分工
- 分析 Agent：中文指令 → 中文字段名+英文值（人可读，值可直接进英文提示词）
- 编译器：纯英文（生图提示词必须英文）
- 抽取器：中文一句话
- QA/返修报告：中文（给人看）
- 需求卡：中文（用户填写界面）
- AI 生成图内文字必乱码 → **禁止文字占位**，文字后期排版加。

## 三、商用规则库（十条，编译器与 QA 必须内置）

| # | 规则 | 落点 |
|---|---|---|
| R1 | Logo/印花几何锁：逐字转写、位置比例锁定；负向 distorted/warped/stretched logo, misspelled text | 商品锁 + 编译器 + QA |
| R2 | 肤色受光真实度：锁肤色深浅，深肤禁提亮、浅肤禁美黑，场景色温下自然渲染 | 人物锁 + 编译器 + QA |
| R3 | 白色过曝+深色死黑双保护：白衣保纹理，深衣保褶皱结构 | 编译器 + QA |
| R4 | 图案对花连续：条纹/格纹在缝线、门襟、口袋处走向连续 | 商品锁 + 编译器 + QA |
| R5 | 面料物理行为：垂坠/挺括/弹力贴身由商品锁判定并注入每组提示词 | 商品锁 + 编译器 |
| R6 | 身材真实性（合规）：按需求如实出图，禁自动瘦身/拉腿/磨骨 | 人物锁 + 编译器 + QA |
| R7 | 手部肢体完整性：负向词常驻 malformed hands, extra fingers, twisted limbs | 编译器 |
| R8 | 背面结构一致性：背面位不查人脸，查发型+体型连续性 | 编译器 + QA |
| R9 | 输出规范：电商主流比例（3:4），主商品完整不裁切，首图留角标安全区 | 生图卡参数 + 编译器 |
| R10 | 抗AI感全套：手机直拍观感、自然曝光、生活痕迹；禁影楼灯/强虚化/磨皮/网红脸；N组表情站位朝向不重复 | 编译器 |

> 裂变到新类目时：十条全保留，再按类目问卷补充类目特有规则（如泳装的水渍反光、羽绒服的充绒蓬度）。
