# 裂变流程 —— 从通用骨架到新类目定制工作流

> Direction 层。接到新类目/新品牌时按此流程执行。每步有产出物和验收标准，做不到就停下问用户。

## 第 0 步：建项目骨架（DBS）
```
新项目/
├── direction/架构总设计.md      ← 记录本次裂变的所有决策与理由
├── blueprints/workflow_spec.json + prompts/*.txt
└── solutions/build_*.py / apply_*.py
```
**最快路径**：复制参考实现 `完全体/` 的 blueprints 和 solutions 作为起点，而不是从零写。
✅ 验收：三层目录就位，构建器能跑通参考实现原样。

## 第 1 步：问卷（禁止跳过）
拿 `blueprints/定制问卷.md` 逐项问用户，把答案写进 direction/架构总设计.md。
**重点是硬指令字段清单**：这个类目哪些属性是图看不出来的？（每漏一项 = 未来一次实测翻车）
✅ 验收：问卷全部有答案；硬指令字段清单经用户确认。

## 第 2 步：定分镜表与槽位协议
- 按类目主图规范排 N 镜（默认8）：标准位在前、生活位在后，必含背面/反面结构位。
- 定槽位语义表（槽1=什么图……），写进 direction。
✅ 验收：用户确认分镜表（这是商用交付物的直接形状，必须用户点头）。

## 第 3 步：只改 Blueprints（裂变的核心动作）
按定制差异从小到大依次改：
1. 需求卡①② 默认值（品类、输出规格、硬指令字段默认值）
2. A1 商品锁字段（类目特有字段，如羽绒服加"充绒蓬度"）
3. A3 分镜表（第 2 步的结果）
4. C 编译器的类目规则段（十条商用规则保留 + 类目特有规则追加）
5. QA 的检查维度与硬失败条款（与硬指令字段一一对应）
6. spec 的输入卡数量/槽位连线（若输入图语义变了）

**禁改清单（除非用户明确要求）**：骨架拓扑、运行组结构、抽取器提示词、槽位协议机制、QA 输出结构（PASS_SUMMARY/评分表/返修简报的骨架）。

✅ 验收：构建器一次编译通过；核验脚本报告蓝图↔产物零漂移。

## 第 4 步：小成本试跑（省钱顺序，🔒 不可乱）
1. 跑组①②（便宜的 LLM 调用）→ 人工检查三张锁卡：字段是否如实、硬指令字段是否正确覆盖（例：A1 是否输出 compression fit 而不是从平铺图猜的 relaxed）。
2. 锁卡不对 → 改需求卡或 prompts → 重跑②。**锁卡不过关禁止跑③**。
3. 锁卡过关 → 跑③④（生图+质检，计费大头，跑前向用户确认）。
✅ 验收：QA 报告可用率 ≥ 目标（默认85%）。

## 第 5 步：按 QA 迭代
- 个别图失败 → 组⑤返修支线单卡重跑（省钱）。
- 系统性失败（多图同一问题）→ 回第 3 步改对应蓝图层（失败在商品保真→A1/C；构图→A3/C；认图错乱→QA）。
- **每次改动走蓝图重建，禁止直接改画布**（content-only 单卡热更新除外，见铁律）。

## 第 6 步：沉淀
- 本次裂变的决策与踩坑写回 direction/架构总设计.md。
- 类目通用的新规则（被第二次验证的）提炼回本框架（硬规则铁律/商用规则库/问卷追加问题）。
- 更新记忆库三份文件，git 提交推送。
✅ 验收：换台电脑 clone 仓库能完整重建交付文件。

## 常见失败模式速查（低模型自检表）

| 症状 | 根因 | 回到哪一步 |
|---|---|---|
| 出图版型/材质不对但图看不出原因 | 硬指令字段缺失或断链 | 第 1 步补问卷 → 第 3 步四级贯通 |
| N 张图之间人物/场景漂移 | 编译器单点一致性被破坏（各分镜自行改写） | 检查 P 是否纯抽取、C 是否一次产全部 |
| QA 报告大量误报 | QA 按序号认图 | QA 改按内容认图（铁律3） |
| 图与提示词槽位对不上 | 连线序≠前缀声明序 | 检查构建器校验是否被绕过 |
| 平台报错无法运行 | 拓扑成环 | 反馈改支线（铁律5） |
| 改了提示词但产物没变 | 直接改了画布没回写蓝图，重建时被覆盖 | 铁律2，蓝图唯一真源 |
