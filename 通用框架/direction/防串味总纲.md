# 防串味总纲 —— 四类串味的机制与护栏（最高优先级文档）

> Direction 层。串味 = 不该出现的内容/规则/记忆出现在了产出里。商用致命，因为客户看到的是"不可控"。
> 本纲把项目里实测过的所有串味按传染机制分四类，每类给护栏清单。**接任何新单前先读本文。**

## 第一类：内容串味（单次产出内混入非本单内容）
**症状**：图里出现旧场景、别的款式元素、没人要求的鞋/饰品、图案少了一个元素。
**机制**：生图模型对"没被明确锁定的可见内容"会自由采样——参考图锚不住的、提示词没点名的，全靠模型先验补，而先验=平均值=串味。
**护栏（全部已实锤验证）**：
1. 输入语义=所见即所得：你给什么就画什么，禁止AI自主选品（V9.1砍S2A/S2B）。
2. 槽位协议程序保证：连线序=槽位序=前缀声明序，构建器校验（铁律1/8）。
3. 场景句逐字锁：分析官产一句场景描述，N组提示词verbatim复用；风格条款按场景类型双模式（R5）。
4. 可见部位枚举：跨图一致性条款枚举所有可见部位，无参考的部位用"中性描述+逐字复用"兜底（R6，鞋子实锤）。
5. 枚举强制令：编译器复述锁卡禁概括措辞，图案逐元素逐色带数量点名——"保持原样"类空话=元素丢失（R11，蓝花实锤）。
6. 定向路由：特殊参考图只连需要它的节点（背面图只进G2/A1/QA），防参考图过载（V9.2）。
7. 真源优先：细节参考用原片像素裁剪（make_detail_crop.py），AI复刻只做兜底（R10）。

## 第二类：版本串味（升级后旧内容/旧问题复活）
**症状**：明明修过的问题又出现；用户以为在测新版实际跑的旧版。
**机制**：两条传染路径——①运行平台上的版本与蓝图脱节（改了没发布/发布没核验）；②修复条款在后续迭代中被无声挤掉。
**护栏**：
1. 蓝图唯一真源+git（铁律2）；禁止画布改动不回写。
2. 发布闭环四步（铁律16）：重建→核验零漂移→发布画布（content热更/结构新建）→回读再核验。
3. 回归防线登记表（铁律17）：每个修复三点落地（条款+QA项+清单项）并登记；改提示词前对照登记表。
4. 结构改动禁原地升级（铁律10），禁止在旧画布项目上测新版本。
5. 核验脚本三件套（verify_lxf/verify_canvas+构建器内置校验）：程序把关，不靠模型自觉。

## 第三类：产品线串味（A品牌/品类的规则渗进B）
**症状**：给新品牌出图带着旧品牌的风格默认值；改一条线的护栏把另一条线改坏。
**机制**：多产品线共用文件时边界不清；品牌特征写死在规则引擎里而不是数据卡里。
**护栏（铁律21）**：
1. 品牌/风格特征只允许存在于数据层（需求卡默认值、A官字段、分镜表），规则引擎（C/QA骨架）保持品牌无关。
2. 每条产品线：独立spec文件+独立交付文件名（版本号入名）+画布项目名带版本；素材按`素材X/用途`目录规范归档。
3. 产品线登记表（回归防线.md内）：登记每条线用哪份spec、哪些文件共用——**修共用文件前先查表，改完对每条线跑回归构建**。
4. 变体禁止复制规则引擎（分镜表数据化，C/QA单份，V10确立）；确需独立引擎（输入协议不同，如V11），在登记表标注"独立护栏，修复需对照移植"。
5. 新品牌必走《定制问卷》，禁止拿旧品牌默认值直接开跑（V9.2版型默认值不匹配实锤）。

## 第四类：记忆串味（经验层面的污染）
**症状**：未验证的猜测被当成经验引用；A项目的特例被当成通用规律用到B。
**护栏**：
1. 三问原则+知识库晋升制：一次验证留项目踩坑，二次验证（或Joe特批）才进40-知识库，条目必须带出处。
2. 踩坑记录里假设必须标注"★待验证"；被推翻的结论要显式改写（V11"生成方差"误判被纠正的处理方式）。
3. 分析节点UNCERTAIN铁律（铁律14）：拿不准写UNCERTAIN，一个猜错的字段会被复制到N张图。
4. 收工必须更新记忆+git推送，异地两机以GitHub为准（双仓库同步）。

## 串味应急诊断流程（任何一类发作时）
1. 拉运行数据（graph/锁卡/编译输出/成品），**从上游往下游找第一个断点**——锁卡对不对→编译丢没丢→参考图挂对没挂对→生成层方差。
2. 断在哪层修哪层，修完走发布闭环+回归登记。
3. 禁止"整体感觉不对就全链路调参"——每次只动定位到的层（V9.2诊断、V11.1蓝花两次实证此流程有效）。
