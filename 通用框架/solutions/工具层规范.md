# 工具层规范 —— 构建器/apply 脚本强制要求 + 交付验收清单

> Solutions 层。工具是"程序保证一致性"的执行者。参考实现：`完全体/solutions/build_lxf.py`、`apply_to_app.py`。

## 一、构建器（蓝图 → 平台文件）强制要求

1. **只读蓝图**：内容全部来自 spec + prompts/*.txt + blueprints/assets/；代码里不许藏提示词或结构。
2. **内置校验（缺一不可）**：
   - ref 唯一性
   - 连接端点存在性
   - 无环（拓扑排序计数 == 节点数，铁律5）
   - 媒体资产存在性（assets 内，铁律12）
   - 打包后回读自检（卡数/连接数/媒体与 manifest 一致）
3. **连接数组顺序 = spec 声明顺序**：绝不排序、绝不去重后重排——顺序就是槽位（铁律1）。
4. **相对路径定位**：`HERE = Path(__file__).resolve().parent`，输出到仓库内相对路径；唯一允许的绝对路径是平台安装路径，且必须是文件头部配置常量。
5. Windows 中文控制台：脚本自带 `sys.stdout.reconfigure(encoding='utf-8')`，用 `-X utf8` 运行。

## 二、核验脚本（独立于构建器的第二双眼睛）

构建 ≠ 验证。必须有独立脚本从蓝图推导期望态，与产物实际态比对：
- 卡数 / 每卡 content / 类型 / AI 参数
- 连接全序列（按标题对逐条比对）
- 分组
输出"零漂移"结论才算交付（铁律2 的验证方法）。

## 三、apply 脚本（蓝图 → 在线画布）强制要求

1. 凭据动态读取：从平台 `data/agent/agent.json` 读 port+token，不硬编码。
2. 调用前 `GET /ping` + `/manifest` 确认登录态与能力（铁律15）。
3. 字段白名单意识：data 只发 content/model/provider/size/resolution/quality；越界整体拒绝；不含 imageUrl（图输入卡搭好后人工放图）。
4. 结构性改动 → 新建项目；老项目只做 content-only 单卡热更新（铁律10/11）。
5. apply 免费幂等，run 计费——脚本永不自动 run，run 必须用户显式触发（铁律6）。
